A well-written workflow catches the cancellation as a `CancelledFailure` (use `isCancellation(err)` to detect it), runs cleanup inside a `nonCancellable` scope, and then rethrows so the workflow execution is recorded as `CANCELED` on the server. ## Key design decisions

### `src/activities.ts`
- `doWork` loops 120 Ă— 500 ms (~60 s total). Each iteration calls `heartbeat()` **before** the `sleep(500)` call. `sleep()` from `@temporalio/activity` is cancellation-aware â€” when the activity is cancelled it throws `CancelledFailure`, which is caught, logged, and rethrown so the worker delivers the cancellation to the workflow.
- `cleanup` returns the literal `"cleanup-done"`.

### `src/workflows.ts`
- `doWork` is proxied with `startToCloseTimeout: '120 seconds'`, `heartbeatTimeout: '10 seconds'`, and `cancellationType: ActivityCancellationType.TRY_CANCEL` â€” this lets the workflow proceed to the cleanup branch immediately without waiting for the activity to gracefully acknowledge cancellation.
- The canonical pattern is applied:
  ```ts
  try {
    await CancellationScope.cancellable(() => doWork());
  } catch (err) {
    if (isCancellation(err)) {
      await CancellationScope.nonCancellable(() => cleanup()); // survives cancellation
      throw err;  // records execution as CANCELED on server
    }
    throw err;
  }
  ```

### `src/worker.ts`
- Uses `NativeConnection.connect({ address, tls: true, apiKey })` (required for Temporal Cloud).
- Prints `WORKER_READY` to stdout once `worker.run()` is entered, so the orchestrator can gate the client start.

### `src/client.ts`
- Uses `Connection.connect({ address, tls: true, apiKey })` and `new Client({ connection, namespace })`.
- Workflow ID is `cancel-wf-${ZEALT_RUN_ID}`.
- After `handle.cancel()`, `handle.result()` is awaited inside a try/catch. The error is confirmed via `err instanceof WorkflowFailedError && isCancellation(err.cause)`, then `cleanup-done observed` is printed to stdout.

### `src/start.ts` (orchestrator)
- Spawns `worker.ts` as a child process, buffers its stdout, and resolves a promise as soon as the `WORKER_READY` token appears (up to 60 s timeout).
- Then spawns `client.ts`, forwarding its stdout directly (so `cleanup-done observed` appears unprefixed in the combined output).
- After the client exits, sends `SIGTERM` to the worker, waits for it to exit (force-kills after 10 s), and calls `process.exit(clientExitCode)` â€” which is `0` on success.
