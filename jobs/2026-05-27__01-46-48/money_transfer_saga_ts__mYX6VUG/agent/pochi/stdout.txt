## Key design decisions

### `workflows.ts` — Saga compensation
- Two `proxyActivities` calls with **different** retry policies:
  - `withdraw` / `refund`: `maximumAttempts: 3` (safe to retry)
  - `deposit`: `maximumAttempts: 1` — ensures the failure path terminates immediately without repeated retries
- Classic saga pattern:
  ```ts
  await withdraw(fromAccount, amount);
  try {
    await deposit(toAccount, amount);
  } catch (err) {
    await refund(fromAccount, amount);   // compensation
    throw err;                           // propagate → FAILED status
  }
  ```

### `activities.ts` — File I/O
- All three activities (`withdraw`, `deposit`, `refund`) use a single `mutateAccounts()` helper: read → mutate → write `/workspace/accounts.json`.
- `deposit` throws `ApplicationFailure.nonRetryable(...)` when `account === "B_FAIL"`, which combined with `maximumAttempts: 1` makes the workflow fail quickly without exhausting retry budgets.

### `run.ts` — Single-process orchestration
- Creates a `NativeConnection` + `Worker` and starts `worker.run()` **without awaiting** (background loop).
- Waits 1.5 s for the worker to start polling, then creates a `Client` and runs both workflows sequentially.
- Catches `WorkflowFailedError` on the second workflow — prints the failure reason and **does not rethrow**, so the script exits with code `0`.
- Calls `worker.shutdown()` + awaits `workerRunPromise` for a clean shutdown before exit.

### Expected final state of `/workspace/accounts.json`
| Account | Start | After saga-ok (−30/+30) | After saga-fail (−50 then +50 refund) | Final |
|---------|-------|------------------------|---------------------------------------|-------|
| A       | 100   | 70                     | 70 (−50 then +50 = net 0)             | **70** |
| B       | 0     | 30                     | (untouched)                           | **30** |
| B_FAIL  | —     | —                      | deposit never ran; no write           | **0** (key added by refund target only if present; key is absent unless deposit ran — actual value depends on whether B_FAIL key existed) |

> Note: `B_FAIL` is only written to the file if `deposit("B_FAIL", ...)` reaches the write — it never does (throws before `mutateAccounts`), so the key will not appear in the file unless it was pre-seeded. Accounts `A=70` and `B=30` are the meaningful post-run values.
