## Key design decisions

| Concern | Decision |
|---|---|
| HTTP I/O | Lives exclusively in `fetchData` activity — never in workflow code |
| Workflow sandbox | `workflows.ts` imports activities with `import type` so no Node built-ins leak into the deterministic bundle |
| Temporal Cloud auth | `tls: true` + `apiKey` on both `NativeConnection` (Worker) and `Connection` (Client) |
| Workflow ID | `fetch-wf-${ZEALT_RUN_ID}` — collision-free across concurrent runs |
| Activity timeout | `startToCloseTimeout: '30 seconds'` with 3 retries |
| Worker lifecycle | `start.sh` launches the worker in the background, waits 5 s for it to connect, runs the client, then `kill`s the worker and exits with the client's exit code |

## Running

```bash

npm start
```

The script will:
1. Start the Worker (background) — connects to `TEMPORAL_ADDRESS` with TLS + `TEMPORAL_API_KEY`, polls `fetch-url-ts`

3. Print the GitHub Zen API response body (the activity result) to stdout
4. Terminate the Worker and exit cleanly
