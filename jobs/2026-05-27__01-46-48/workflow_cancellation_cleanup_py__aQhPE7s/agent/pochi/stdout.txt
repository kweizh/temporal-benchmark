# Graceful Workflow Cancellation with Shielded Cleanup (Temporal Python SDK) ## Background Temporal Workflows can be canceled by clients. In Python, a cancellation is delivered to the workflow as an `asyncio.CancelledError` raised at the next `await` point. A well-written workflow can catch this exception, run any required cleanup (release locks, free resources, write audit logs), and then re-raise the error so the workflow is recorded as `CANCELED` on the server. Because cleanup work also runs inside the cancelled task, it must be **shielded** (e.g. with `asyncio.shield`) so the cancellation does not abort the cleanup itself. In this task you will implement this pattern end-to-end against **Temporal Cloud** using the **Python SDK**. ## Requirements - Implement a Temporal Activity `release_resources(job_id: str)` in Python that appends a single line `released:<job_id>` to the file `/workspace/cleanup.log` (creating the file if necessary). - Implement a Temporal Workflow `LongJobWorkflow` in Python that: - Takes a single argument `job_id: str` in its `@workflow.run` method. - Calls `await workflow.sleep(timedelta(minutes=10))` as its main work. - When cancellation is requested, catches `asyncio.CancelledError`, executes the `release_resources` activity for the given `job_id` in a way that survives cancellation (i.e. shields the activity invocation so it completes even though the workflow task itself has been cancelled), and then re-raises `asyncio.CancelledError` so the workflow ends in the `CANCELED` state. - Implement a Worker that polls the task queue `cancel-py` and hosts both `LongJobWorkflow` and `release_resources`. - Implement a Client script that: 1. ## What each file does

### `activities.py` â€” `release_resources(job_id)`
Decorated with `@activity.defn`. Appends `released:<job_id>` to `/workspace/cleanup.log` (creates the file if absent).

### `workflows.py` â€” `LongJobWorkflow`
Decorated with `@workflow.defn`. Its `run(job_id: str)` method:
1. Calls `await workflow.sleep(timedelta(minutes=10))` as the main work.
2. Catches `asyncio.CancelledError` in an `except` block.
3. Inside that block, runs the cleanup activity **shielded**:
   ```python
   await asyncio.shield(
       workflow.execute_activity(
           release_resources, job_id,
           start_to_close_timeout=timedelta(seconds=30),
       )
   )
   ```
   `asyncio.shield` prevents the already-cancelled workflow task from propagating the cancellation into the activity coroutine, so the activity always runs to completion.
4. Re-raises `CancelledError` so the workflow ends in `CANCELED` state on the server.

### `worker.py` â€” Worker
Polls task queue `cancel-py` hosting both `LongJobWorkflow` and `release_resources`. Connects to Temporal Cloud via `TEMPORAL_ADDRESS`, `TEMPORAL_NAMESPACE`, `TEMPORAL_API_KEY` with `tls=True`.

### `client.py` â€” Client
Starts `LongJobWorkflow` with id `job-${ZEALT_RUN_ID}` and `job_id="alpha"`, waits 3 s, calls `handle.cancel()`, then awaits `handle.result()`. Catches `WorkflowFailureError` whose cause is `temporalio.exceptions.CancelledError` and exits 0.

### `run.py` â€” Orchestrator
