## Key Design Decisions

### Fan-out / Fan-in pattern (`workflows.ts`)
All activity invocations are built as an array of promises via `.map()` **without awaiting individually**, then awaited together with `Promise.all(...)`. This causes Temporal to schedule all `squareNumber` activity tasks simultaneously — satisfying the requirement that all `ActivityTaskScheduled` events appear within a small time window.

```typescript
const squarePromises = numbers.map((n) => squareNumber(n)); // fan-out (no await)
const squares = await Promise.all(squarePromises);           // fan-in
return squares.reduce((acc, val) => acc + val, 0);           // sum = 55
```

### Cloud connection
Both the Worker (`NativeConnection.connect`) and Client (`Connection.connect`) use `tls: true` + `apiKey` from `TEMPORAL_API_KEY`, reading `TEMPORAL_ADDRESS` and `TEMPORAL_NAMESPACE` from the environment. No credentials are hard-coded.

### Workflow ID
`parallel-wf-${process.env.ZEALT_RUN_ID}` — unique per run, collision-free.

### `npm start`
`start.ts` spawns `dist/worker.js` as a background child process, waits 5 s for it to connect and begin polling, then executes the workflow inline via the client, prints the result, sends `SIGTERM` to the worker, and exits.
