# Temporal Workflow with Parallel Activities via `asyncio.gather` (Python) ## Background You are building a Temporal application with the Temporal.io Python SDK (`temporalio`). The goal is to demonstrate the canonical pattern for fan-out parallelism inside a Temporal **Workflow**: invoking many **Activities** concurrently with `asyncio.gather` and aggregating their results. Each Activity issues a single `GET` request to a URL and returns its HTTP status code; the Workflow then returns a mapping of URL to status code. Temporal Cloud credentials are already provided to the environment as the following variables: `TEMPORAL_API_KEY`, `TEMPORAL_ADDRESS`, and `TEMPORAL_NAMESPACE`. Do **NOT** hard-code or invent credentials; read them from the environment. The unique run identifier is provided as `ZEALT_RUN_ID`. ## Requirements * Implement a Python Temporal project under `/home/user/myproject` that contains: * An **Activity** named `fetch_url` that accepts a single string argument `url`, uses `httpx.AsyncClient` to issue `GET <url>`, and returns the HTTP status code as a Python `int`. * A **Workflow** named `ParallelFetchWorkflow` that accepts a single argument `urls` (a list of strings), schedules one `fetch_url` activity per URL **concurrently** using `asyncio.gather`, and returns a `dict[str, int]` mapping each input URL to its HTTP status code. * A **Worker** that connects to Temporal Cloud using the API key, registers `ParallelFetchWorkflow` and `fetch_url`, and polls the task queue `parallel-py`. * A **Client** entrypoint that connects to Temporal Cloud, executes `ParallelFetchWorkflow` with the URL list `["https://www.example.com", "https://api.github.com", "https://httpbin.org/status/200"]`, waits for completion, and prints the resulting dictionary to stdout. * The Worker and Client must use the task queue `parallel-py`. * The Workflow ID must be exactly `parallel-${ZEALT_RUN_ID}` where `${ZEALT_RUN_ID}` is read from the `ZEALT_RUN_ID` environment variable so concurrent runs do not collide. * Provide a runnable entrypoint that starts the Worker and the Client together against Temporal Cloud (for example, a single `python main.py` command, or any `python -m` style entrypoint that does the same). After the workflow completes, the printed dictionary must contain each of the three input URLs as a key, each mapping to integer `200`. ## Implementation Hints * Install the official Temporal Python SDK with `pip install temporalio` and the HTTP client with `pip install httpx`. * For Temporal Cloud, connect with `temporalio.client.Client.connect(address, namespace=..., api_key=..., tls=True)`. * Define the activity using `@activity.defn` from `temporalio` and the workflow using `@workflow.defn` plus `@workflow.run` on the async `run` method. * Inside the workflow, build a list of awaitables by calling `workflow.execute_activity(fetch_url, url, start_to_close_timeout=timedelta(seconds=30))` for each URL **without** awaiting individually, then await them all together with `asyncio.gather(*tasks)` so the activities are scheduled in parallel. * After `asyncio.gather` returns, zip the input URL list with the result list into a `dict[str, int]` and return it. * The activity must use `httpx.AsyncClient` to perform the GET request and return `response.status_code` as `int`; do not call `requests` or any synchronous HTTP client inside an async activity. * Use `temporalio.worker.Worker` to register the workflow and activity on the `parallel-py` task queue, and `Client.execute_workflow` to start `ParallelFetchWorkflow` with the required arguments. * Read `ZEALT_RUN_ID`, `TEMPORAL_API_KEY`, `TEMPORAL_ADDRESS`, and `TEMPORAL_NAMESPACE` from `os.environ`. * You may run the worker and the client in the same `asyncio` event loop (for example by starting the worker as a background task, executing the workflow, then cancelling the worker) — just make sure the workflow completes against Temporal Cloud and the result is printed before the process exits. ## Acceptance Criteria * Project path: /home/user/myproject * Start command: `python main.py` (executed from `/home/user/myproject`) * Task queue: `parallel-py` * Workflow type name: `ParallelFetchWorkflow` * Activity type name: `fetch_url` * Workflow ID: must be exactly `parallel-${ZEALT_RUN_ID}` using the value of the `ZEALT_RUN_ID` environment variable * When `python main.py` is executed, the workflow `ParallelFetchWorkflow` must be started against Temporal Cloud using the namespace from `TEMPORAL_NAMESPACE`, run to completion, and the resulting dictionary must be printed to stdout. * The completed workflow execution must be visible in Temporal Cloud (status `COMPLETED`) and its result must be a JSON object containing each of the three input URLs (`https://www.example.com`, `https://api.github.com`, `https://httpbin.org/status/200`) as a key, each mapping to the integer `200`. * The activities must be scheduled in parallel, not sequentially: all three `fetch_url` activity-task-scheduled events must appear in the workflow history within a small time window of each other (a few seconds).
