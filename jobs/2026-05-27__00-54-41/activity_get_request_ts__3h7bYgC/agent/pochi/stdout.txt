# HTTP GET Activity Temporal Workflow (TypeScript) ## Background You are building a Temporal application using the Temporal.io TypeScript SDK that demonstrates the canonical pattern of wrapping a non-deterministic operation (an outbound HTTP call) inside an **Activity** while orchestration logic lives in a deterministic **Workflow**. The Worker connects to **Temporal Cloud** using an API key, and a Client starts the Workflow and prints the result returned by the activity. Temporal Cloud credentials are already provided to the environment as the following variables: `TEMPORAL_API_KEY`, `TEMPORAL_ADDRESS`, and `TEMPORAL_NAMESPACE`. Do **NOT** hard-code or invent credentials; read them from the environment. ## Requirements * Implement a TypeScript Temporal project that contains: * An **Activity** named `fetchData` that takes a URL string, performs a real HTTP GET request against that URL (no mocks), and returns the response body as a string. * A **Workflow** named `FetchUrlWorkflow` that accepts a URL string, invokes the `fetchData` activity, and returns the activity's result string. * A **Worker** that connects to Temporal Cloud (via API key + TLS) and polls the task queue `fetch-url-ts`. * A **Client** entrypoint that connects to Temporal Cloud, starts (or executes) `FetchUrlWorkflow` with the input URL `https://api.github.com/zen`, waits for completion, and prints the activity result string to stdout. * The Workflow ID must use the prefix `fetch-wf-` and include the value of the `ZEALT_RUN_ID` environment variable so concurrent runs do not collide (e.g. `fetch-wf-${ZEALT_RUN_ID}`). * The Worker and Client must use the task queue `fetch-url-ts`. * Provide an `npm start` script that runs the Worker in the background, runs the Client, waits for the workflow result, prints it, and then exits cleanly (the Worker may be terminated after the Client finishes). ## Implementation Hints * Install the official TypeScript SDK packages: `@temporalio/client`, `@temporalio/worker`, `@temporalio/workflow`, `@temporalio/activity`. * Activities are the correct place for I/O. Use the global `fetch` available in Node 18+ (or `node-fetch`) inside the activity, not inside the workflow, since network calls are non-deterministic. * For the Client, use `Connection.connect` from `@temporalio/client` with `address`, `tls: true`, and `apiKey` options. For the Worker, use `NativeConnection.connect` from `@temporalio/worker` with the same connection options. * Pass `namespace` to both the `Client` constructor and `Worker.create`. * Read `ZEALT_RUN_ID`, `TEMPORAL_API_KEY`, `TEMPORAL_ADDRESS`, and `TEMPORAL_NAMESPACE` from `process.env`. * Use `proxyActivities` from `@temporalio/workflow` to invoke the activity from the workflow. An Activity must have either `startToCloseTimeout` or `scheduleToCloseTimeout` set; pick a reasonable timeout (e.g., 30 seconds) appropriate for an HTTP call. * You can run the worker in the background of the same `npm start` script (for example with `concurrently`, a shell `&`, or by spawning it from the client script) — just make sure the workflow completes and its result is printed before the script exits. ## Acceptance Criteria * Project path: /home/user/myproject * Start command: `npm start` (executed from `/home/user/myproject`) * Task queue: `fetch-url-ts` * Workflow type name: `FetchUrlWorkflow` * Activity type name: `fetchData` * Workflow ID: must start with `fetch-wf-` and include the value of the `ZEALT_RUN_ID` environment variable * When `npm start` is executed, the workflow `FetchUrlWorkflow` must be started against Temporal Cloud using the namespace from `TEMPORAL_NAMESPACE`, run to completion, and the response body string returned from the `fetchData` activity must be printed to stdout. * The completed workflow execution must be visible in Temporal Cloud (status `COMPLETED`), and its result must be a non-empty string returned by the activity from the real HTTP GET to `https://api.github.com/zen`.
