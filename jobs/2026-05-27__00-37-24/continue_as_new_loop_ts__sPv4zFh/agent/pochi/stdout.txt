# Continue-As-New Loop Workflow (TypeScript) ## Background You are building a long-running counter on top of Temporal Cloud using the Temporal.io TypeScript SDK. A naive workflow that loops thousands of times in one Run Id will eventually accumulate a huge event history and approach Temporal's event history limits. To stay healthy, long loops should periodically call `continueAsNew` so the workflow closes its current Run Id and starts a fresh one with the same Workflow Id, carrying its current state forward. In this task you are given a starter TypeScript Temporal project. The workflow file (`src/workflows.ts`) ships with a **broken** `LongLoopWorkflow` implementation that loops from `start` up to `target` calling the `incrementCounter` activity for each iteration, all in a single Run Id. You must refactor it so that the workflow calls `continueAsNew` every 10 iterations, passing the current counter value forward as the new `start` argument. The workflow should still ultimately complete and return the final counter value (which must equal `target`). Temporal Cloud credentials are already provided to the environment as `TEMPORAL_API_KEY`, `TEMPORAL_ADDRESS`, and `TEMPORAL_NAMESPACE`. Do **NOT** hard-code or invent credentials; read them from the environment. The current run id is provided as `ZEALT_RUN_ID`; use it to scope the workflow id. ## Requirements Implement a TypeScript Temporal project with the following pieces (extending the starter project under `/home/user/myproject`): * A **Workflow** named `LongLoopWorkflow` with the signature `LongLoopWorkflow(start: number, target: number): Promise<number>` that: * Treats `start` as the **current** counter value, and `target` as the value the chain should eventually reach. * On each iteration, invokes the activity `incrementCounter(currentValue)` and updates its local counter to the returned value. * Calls `continueAsNew` every **10** iterations (i.e., after the 10th, 20th, 30th, ... increment within a single Run Id), passing the latest counter value forward as the new `start` and forwarding `target` unchanged. After calling `continueAsNew`, the current Run Id must close successfully and a new Run Id with the same Workflow Id must take over. * Returns the final counter value when the counter reaches `target`. The final returned value must equal `target`. * An **Activity** named `incrementCounter(currentValue: number): Promise<number>` that: * Returns `currentValue + 1`. * Appends a single line containing exactly the new value followed by a newline (`<newValue>\n`) to the file `/workspace/loop.log`. The activity must ensure `/workspace` exists before writing. * A **Worker** that connects to Temporal Cloud (API key + TLS) and polls the task queue `loop-ts`. * A **Client** entrypoint that, against the same Temporal Cloud namespace, starts `LongLoopWorkflow` with arguments `start = 0`, `target = 25`, task queue `loop-ts`, and workflow id `loop-${ZEALT_RUN_ID}`, then awaits the final result (following the continue-as-new chain) and prints it to stdout as `Final result: <number>`. The project must expose an `npm start` script that starts the Worker, runs the Client, awaits the workflow result, prints the final result, and exits cleanly with exit code `0`. ## Implementation Hints * Read the official `continueAsNew` guidance for the TypeScript SDK at https://docs.temporal.io/develop/typescript/continue-as-new . * Import `continueAsNew` (and `proxyActivities`) from `@temporalio/workflow`. The typed form `continueAsNew<typeof LongLoopWorkflow>(start, target)` ensures the arguments line up with the workflow signature. * Calling `continueAsNew` ends the current Run Id; control will not return to the caller in the same run. You do **not** need to `return` after it. * Activities must perform the file I/O (workflows are sandboxed). In `incrementCounter`, ensure `/workspace` exists and append `"<newValue>\n"` to `/workspace/loop.log`. * The client must `await handle.result()` so it follows the continue-as-new chain to the **final** Run Id and reads its result (the SDK does this automatically when you call `result()` on a handle returned by `client.workflow.start`). * Use `Connection.connect` from `@temporalio/client` and `NativeConnection.connect` from `@temporalio/worker` with `address`, `tls: true`, and `apiKey`. Pass `namespace` to both the `Client` constructor and `Worker.create`. * Read `ZEALT_RUN_ID`, `TEMPORAL_API_KEY`, `TEMPORAL_ADDRESS`, and `TEMPORAL_NAMESPACE` from `process.env`. * You may run the worker in the background of the same `npm start` script.
